Understanding JWT Tokens: The Core of Web Authentication

January 25, 2025 · 9 min read

"How does staying logged in work?" That was my question when I first learned web development. HTTP is stateless—how does the server remember me?

First, Understanding Sessions

To understand JWT, you need to know the traditional session approach.

  1. User logs in
  2. Server creates a session ID and stores it in memory/database
  3. Session ID sent to browser as a cookie
  4. Cookie automatically sent with every request
  5. Server looks up user info by session ID

The problem: The server must remember all sessions. Multiple servers? You need session synchronization. That's complicated.

JWT: The Server Doesn't Need to Remember

JWT (JSON Web Token) takes a different approach. Instead of the server storing user info, it puts the information in the token itself and gives it to the client.

The token includes the server's signature, so later when it receives the token, it can verify "yep, I made this."

JWT Structure: Three Parts Separated by Dots

A JWT looks like this:

xxxxx.yyyyy.zzzzz

Header.Payload.Signature

1. Header

Token type and signing algorithm. Base64Url encoded.

{"alg": "HS256", "typ": "JWT"}

2. Payload

The actual data. Called "claims."

{

"sub": "1234567890",

"name": "John Doe",

"admin": true,

"exp": 1706234567

}

⚠️ Important: The payload is NOT encrypted! Anyone can decode it. Don't put sensitive data here.

3. Signature

Created by combining header + payload + secret key.

HMACSHA256(

base64UrlEncode(header) + "." + base64UrlEncode(payload),

secret

)

This signature lets the server detect if the token was tampered with.

JWT Authentication Flow

  1. User logs in
  2. Server creates and returns a JWT
  3. Client stores JWT (usually localStorage or httpOnly cookie)
  4. Subsequent requests include Authorization: Bearer {token} header
  5. Server verifies signature and uses payload info

Sessions vs JWT

Sessions JWT
Storage Server Client
Scalability Needs session sync No server sharing needed
Logout Delete session, done Hard to invalidate token
Security Server-managed Token theft is dangerous

JWT Security Considerations

1. Expiration (exp) is a must

Set an expiration time. Permanent tokens are dangerous. Usually 15 minutes to 1 hour.

2. Use Refresh Tokens

Short-lived access tokens with refresh token rotation is safer.

3. HTTPS is mandatory

Tokens are exposed in plain text over HTTP.

4. localStorage vs httpOnly cookies

localStorage is vulnerable to XSS; cookies to CSRF. httpOnly cookie + CSRF token is safest.

When to Use JWT

  • Microservices architecture (shared auth between services)
  • Mobile apps (instead of cookies)
  • Single Sign-On (SSO)
  • API authentication

For simple websites, sessions might be simpler. Choose the right tool for the job.
